FILE_TITLE: launcher_subsystem_migration_debate정반합 Debate: WKAppBot.Launcher Console→Windows Subsystem🔵 THESIS — Switch is Worth ItCore argument: Console subsystem is the wrong default for a process whose primary job is background orchestration. The flash occurs because Windows allocates a conhost before your code runs a single line. Switching to Windows subsystem eliminates that allocation entirely — no conhost, no flash, no inherited console baggage contaminating child spawns.Gains:

Eye/guardian daemon launches silently with zero user-visible artifact
CREATE_NO_WINDOW on child workers becomes redundant — the launcher itself has no window to inherit
Hot-swap re-launches don't momentarily surface a console to the taskbar
Cleaner separation: launcher is infrastructure, not a REPL
🔴 ANTITHESIS — Serious Risks Block a Naive SwitchMCP stdio is the killer constraint. Claude Desktop / MCP hosts launch the server via stdio and expect the process to be a well-behaved console process. stdin/stdout on a Windows-subsystem process with no attached console returns INVALID_HANDLE_VALUE from GetStdHandle. The .NET runtime's Console.In/Out will throw or silently drop bytes. Your entire MCP proxy relay breaks.Specific failure modes:
AttachConsole(ATTACH_PARENT_PROCESS) works only if the parent actually has a console. When launched by Claude Desktop (a GUI app), there is no parent console → AttachConsole returns false → GetStdHandle(STD_INPUT_HANDLE) is NULL → pipe reads return 0 bytes → MCP server appears to hang.

AllocConsole() after the fact creates a new console, not the inherited stdio handles. The pipe handles passed by the MCP host are already in the process handle table — they are not console handles. AttachConsole/AllocConsole do not retroactively fix them. You must use the raw inherited handles via GetStdHandle before any console manipulation or capture them at startup before touching the console subsystem.
