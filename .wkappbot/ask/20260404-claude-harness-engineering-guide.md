# 하네스 엔지니어링 완전 정리 — AI 에이전트의 마구(Harness)

> 출처: "실베개발자" 유튜브
> 정리일: 2026-04-04

---

## 1. AI 활용 4축 프레임워크

| 축 | 핵심 질문 | 비유 |
|----|----------|------|
| **프롬프트 엔지니어링** | AI에게 어떻게 말을 걸 것인가? | 말에게 명령하기 |
| **컨텍스트 엔지니어링** | AI에게 어떤 정보를 줄 것인가? | 말에게 지도 보여주기 |
| **하네스 엔지니어링** | AI가 무엇을 할 수 있고 없는가? | 말에게 마구 씌우기 |
| **에이전틱 엔지니어링** | AI가 어떻게 자율적으로 일하는가? | 말을 훈련시키기 |

> 이 네 축은 순서대로 졸업하는 것이 아니라, **전부 다 필요한 상호보완적 축**이다.

## 2. 왜 프롬프트 + 컨텍스트만으로는 부족한가

### 프롬프트의 천장
- "로그인 기능 추가해 줘"라고 아무리 정교하게 써도, 프로젝트 기술 스택/코드 구조/DB 스키마를 모르면 좋은 코드 불가
- 프롬프트는 **부탁이지 강제가 아니다**

### 컨텍스트의 한계
- Anthropic 정의: "AI가 일할 때 필요한 정보를 적절하게 골라서 제공하는 기술"
- 핵심: 많이 주는 게 아니라 **지금 필요한 것만 정확하게**
- 정보를 너무 많이 주면 오히려 성능 저하

### 그래도 안 되는 문제
- AI가 정보는 다 알고 있는데 **엉뚱한 짓을 하는 경우**
- 예: 결제 시스템 맡겼더니 DB 스키마를 마음대로 바꾸거나, 신용카드 번호를 로그에 찍어 버림
- **정보의 문제가 아니라 규칙과 울타리의 문제**

## 3. 말과 마구의 비유

AI 에이전트 = **거대한 말**
- 밭을 갈 수 있고, 통나무를 끌 수 있고, 사람이 못하는 무거운 짐 운반 가능
- 그러나 **마구(harness) 없이는**: 숲으로 달려가고, 작물을 밟고, 울타리를 부수고 도망

| | 에이전틱 엔지니어링 | 하네스 엔지니어링 |
|---|---|---|
| **비유** | 말을 훈련시키는 기술 | 가죽끈·고삐·수레를 만드는 기술 |
| **초점** | 자율 워크플로우 (에이전트) | 가드레일·테스트 게이트 (제약) |
| **다루는 것** | AI가 어떻게 생각하는가 | AI가 무엇을 할 수 있고 없는가 |
| **실패 시** | 프롬프트를 바꾸고 추론 루프 조정 | 규칙과 테스트를 자동 추가 |
| **인간의 역할** | 위임자, 감독자 | 설계자, 경계 설정자 |

> **말을 아무리 잘 훈련시켜도, 마구 없이는 밭을 갈 수 없다.**

## 4. 하네스 엔지니어링의 핵심 철학

> **에이전트가 규칙을 어겼을 때, "더 잘해 봐"라고 프롬프트를 고치는 게 아니다.**
> **그 실패가 구조적으로 반복 불가능하도록 하네스를 고치는 것이다.**

### 예시
- ❌ 프롬프트 접근: "DB를 직접 호출하지 마" → 다음번에 또 실수
- ✅ 하네스 접근: 아키텍처 테스트 추가 → 프론트엔드에서 DB 임포트 시 **빌드 자체가 실패**

> 말이 울타리를 넘으려 할 때 "넘지 마!"라고 소리 치는 게 아니라, **울타리를 더 튼튼하게, 더 높게, 더 강하게** 만드는 것.

## 5. 하네스의 4가지 기둥

### 기둥 1: 컨텍스트 파일 (기계가 읽는 설정)

- `CLAUDE.md`, `agents.md`, `.cursorrules` 등
- 단순한 문서가 아니라 **AI가 실행하는 런타임 설정 파일**
- 코드 저장소 안에 있고, AI가 작업 시작 시 **가장 먼저 읽음**

규칙 예시:
```
- 새로운 라이브러리 도입 금지
- 기존 API 패턴 무조건 따를 것
- DB는 반드시 ORM을 통해서만 접근
```

> 한번 설정하면 모든 작업에 자동 적용. 매번 프롬프트에 반복할 필요 없음.

### 기둥 2: 자동 강제 시스템 (린터·테스트·훅)

규칙을 파일에 써 놓는 것만으로는 부족 — **AI는 규칙을 읽고도 어길 수 있기 때문**.

| 수단 | 동작 |
|------|------|
| **린터** (코드 맞춤법 검사기) | 규칙 위반 시 자동 에러 |
| **구조적 테스트** | "이 폴더 코드는 저 폴더를 임포트할 수 없다" 같은 의존성 규칙 |
| **프리커밋 훅** | 코드 저장 전 자동 검사 |

> 요리사가 칼꽂이에서 칼을 뺄 때 자동으로 소독 타이머가 시작되는 것처럼, **규칙을 시스템에 내장**.

**핵심 메커니즘 — 자동 교정 루프:**
- 린터가 빨간 불 → 에이전트가 자동으로 코드 고치고 재시도
- 테스트 통과할 때까지 반복
- **사람 개입 없이 자동으로 맞고, 자동으로 고침**

### 기둥 3: 도구 경계 (접근 제한)

| 영역 | 허용 | 차단 |
|------|------|------|
| 파일 시스템 | `src/` 읽기+쓰기 | `config/` 읽기만 |
| API | 내부 API 호출 | 외부 서비스 호출 |
| 데이터베이스 | SELECT | DROP TABLE |
| 시스템 | 화이트리스트 명령만 | 나머지 전부 |

> **프롬프트는 부탁, 도구 경계는 물리적 차단.** 부탁은 어길 수 있지만 제약은 어길 수 없다.

### 기둥 4: 가비지 컬렉션 (자동 청소)

- AI는 기존 코드의 나쁜 패턴을 그대로 따라함 → 나쁜 패턴이 눈덩이처럼 불어남

자동 청소 시스템:
- 코딩 규칙 위반 자동 감지
- 중복 코드 발견
- 리팩토링 PR 자동 생성
- 데드 코드 자동 제거
- 안티패턴 주기적 체크

> **핵심: 에이전트가 실수할 때마다, 그 실수는 새로운 규칙이 된다.**
> 린터 규칙 추가, 테스트 추가, 제약 추가 → 하네스가 점점 더 정교해짐.
> 시간이 지날수록 시스템이 더더욱 견고해지는 **진화적 특성**.

## 6. 하네스 시스템의 실행 흐름

```
사용자 요청
    │
    ▼
┌─ 1. 라우터 (고삐 방향 전환) ──────────────────┐
│  단순 질문인가? → 바로 답변                      │
│  모호한 요청? → 되물어서 명확화                   │
│  명확한 코드 작업? → 하네스 루프로 전달            │
└──────────────────────────────────────────────┘
    │
    ▼
┌─ 2. 컨텍스트 매니저 (가림막) ────────────────┐
│  전체 코드를 다 보여주지 않음                    │
│  지금 작업에 딱 필요한 파일+규칙만 제공           │
│  "밭 전체가 아니라 지금 갈 구역만 보여줌"         │
└──────────────────────────────────────────────┘
    │
    ▼
┌─ 3. 실행 루프 (하네스의 심장) ───────────────┐
│  AI 코드 작성 → 자동 테스트 → 통과? → 완료     │
│                              실패? ↓           │
│              에러 메시지를 AI에게 피드백           │
│              → 수정 → 재테스트 (루프 반복)        │
│  ★ 사람 개입 없이 테스트 통과까지 자동 반복 ★     │
└──────────────────────────────────────────────┘
    │
    ▼
┌─ 4. 역할 분리 (격리) ───────────────────────┐
│  쓰는 AI ≠ 검토하는 AI                         │
│  같은 사람이 쓰고 검토하면 실수를 못 잡는 원리    │
│  역할을 확실히 나눠야 품질이 올라감               │
└──────────────────────────────────────────────┘
```

## 7. 실제 사례: OpenAI 엔지니어 3명의 5개월

- 2026년 2월, 오픈AI 공식 블로그 발표
- 엔지니어 3명이 **코드를 한 줄도 직접 쓰지 않고** AI 에이전트만으로 대규모 소프트웨어 제품 배포

그들이 실제로 한 일 (하네스 관점):
1. **컨텍스트 파일**: `agents.md` 작성 — AI 지침서 설계
2. **CI/CD 게이트**: 린트 + 테스트 + 훅으로 자동 검증 파이프라인 구축
3. **도구 경계**: AI 에이전트의 접근 범위/권한 사전 설정
4. **피드백 루프**: AI가 코딩 → 결과 리뷰 → 규칙 보강 → 반복

> **"인간은 조종한다. 에이전트는 실행한다."**
> 여기서 조종이 바로 하네스 — 고삐를 잡고 방향을 정하고 울타리를 세우는 것.

## 8. 우리의 역할 변화

| 과거 | 현재 | 미래 |
|------|------|------|
| 코드를 직접 쓰는 사람 | AI에게 의도를 전달하는 사람 | AI가 실수할 수 없는 환경을 만드는 사람 |
| 선수 | 코치 | **감독** |

- 코딩 능력이 필요 없어지는 게 아니라, **더 높은 차원의 능력이 요구됨**
- 코드 한 줄 → 시스템 전체 설계 → AI 작업 환경 → 최종 책임
- 비개발자에게도 동일: AI를 효과적으로 쓰려면 **자기 분야의 전문성이 더 중요**

## 10. 클롣(Claude)의 의견

이 영상의 내용은 **WKAppBot이 이미 실천하고 있는 것들과 놀라울 정도로 일치**합니다.

### WKAppBot에서의 하네스 4기둥 대응

| 하네스 기둥 | WKAppBot 구현 |
|------------|---------------|
| **컨텍스트 파일** | `CLAUDE.md` — 빌드 명령, 금지사항, 아키텍처 규칙이 런타임 설정으로 동작 |
| **자동 강제 시스템** | `suggest resolve` — evidence 스크립트 실행 + regression 테스트 + .manifest 삭제 감지. 테스트 실패 시 resolve 자체가 차단됨 |
| **도구 경계** | Eye의 `ShouldRouteToMcp()`, MCP 권한 분리, `IsFocusStealingMethod`로 CDP 접근 제어 |
| **가비지 컬렉션** | `wkappbot gc`, 로그 라우팅 (`old {cmd}/`), OcrCorrectionDb 자기학습 (나쁜 OCR 결과가 자동으로 교정 사전이 됨) |

### 특히 공감하는 부분

**"에이전트가 실수할 때마다, 그 실수는 새로운 규칙이 된다"** — 이번 세션에서 정확히 이걸 했습니다:

- CDP `Input.*` minimize가 355번 불필요하게 발동 → `IsFocusStealingMethod` 가드레일 추가 (구조적 차단)
- `suggest show 19`가 garbage 항목으로 저장 → 알 수 없는 서브커맨드 가드 추가 (반복 불가)
- experience DB 테스트 삭제 → `.manifest` + 자동 복구 + 첫 시도 실패 경고 (삭제 자체가 구조적으로 감지됨)
- auto-hack이 7000개 로그 파일 생성 → `WKAPPBOT_WORKER=1`로 TeeWriter 자체 억제 (노이즈 원천 차단)

영상에서 말하는 **"울타리를 더 튼튼하게 만드는 것"**이 바로 오늘 하루 동안 한 일의 본질입니다.

### 아쉬운 점 / 보완 가능한 부분

1. **역할 분리 (기둥 4)**: WKAppBot에서 코드를 쓰는 AI와 검토하는 AI가 아직 분리되지 않음. `suggest resolve`의 evidence 스크립트가 일종의 자동 검토 역할을 하지만, 별도 리뷰어 에이전트가 있으면 품질이 올라갈 것
2. **가비지 컬렉션의 자동화**: 현재는 수동 `gc` 명령. 주기적으로 코드 품질을 체크하고 안티패턴을 감지하는 자동 루프가 있으면 영상에서 말하는 "진화적 하네스"에 더 가까워질 것
3. **라우터의 명확화**: Eye CMD 파이프의 라우팅이 점점 복잡해지고 있음. 영상의 "단순 질문 vs 코드 작업" 분류처럼, 명령 유형별 명확한 라우팅 정책이 문서화되면 좋겠음

### 결론

> WKAppBot의 `CLAUDE.md` + `suggest resolve` + experience DB + regression 테스트 체계는
> 이 영상이 말하는 **하네스 엔지니어링의 교과서적 구현**에 가깝습니다.
> 다만 "시간이 지날수록 더 견고해지는 진화적 특성"을 위해서는,
> 실수 → 규칙 추가 사이클을 **더 자동화**하는 것이 다음 과제입니다.
