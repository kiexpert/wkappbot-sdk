# Anthropic Claude Code 유출 사태 분석

> 출처: "안될과학" 유튜브 — AI 시대의 총화 김태훈 과학 커뮤니케이터 인터뷰
> 정리일: 2026-04-04

---

## 1. 사건 개요

- **Anthropic의 Claude Code 내부 코드 약 51만 줄이 유출**
- 며칠 전 내부 필드 파일이 드러난 데 이어, 이번에는 대규모 코드 자체가 유출
- Anthropic 측은 **사고(프로세스 문제)로 인정** — 의도적 유출이 아님
- 자사 직원 보호를 위해 "퍼나르지 말아 달라"는 공지 발표

---

## 2. Anthropic과 Claude

- **대표**: 오픈AI 출신 — 뜻이 안 맞아 독립
- 안전(AI Safety)을 강조하며 설립한 회사
- Claude는 **개발자 중심 사용자층**이 두텁고, 유료 사용량(3만원/30만원 플랜)이 많음
- OpenAI의 ChatGPT에 대응하는 포지션

---

## 3. 하네스 엔지니어링 (Harness Engineering)

### 개념
- 말(horse)이 힘도 세고 강력하지만 **하네스(고삐)로 잘 잡아줘야 적재적소에 쓸 수 있다**는 비유
- 이전에는 "하네스 엔지니어링"이라는 말은 있었지만 실제 적용 여부는 불분명했음
- **유출 코드를 통해 실제로 광범위하게 사용되고 있음을 확인**

### 핵심 발견
- Claude Code는 단순 LLM 서비스가 아닌 **에이전트 운영 체제(Agent OS)에 가까운 구조**
- LLM Wrapper라고 표기되어 있으나, 실제로는:
  - 모델이 "튕겨 나가지 않게" 하는 장치
  - 보안 제어
  - 권한 관리
  - 일탈 방지
- **"모델보다 어쩌면 하네스가 더 낫다"**
  - 모델이 아무리 좋아도 하네스가 잘 안 갖춰져 있으면 사용자 체감 만족도가 낮아짐
  - 프롬프트를 정말 잘 짜면 극복 가능하지만, 대부분의 사용자는 그렇지 않음

## 4. 보안 구조 — 3중+ 방탄 체계

### 4-1. 계층적 보안 (20겹 이상 비유)
- **우선순위 기반 탐색**: 1번에서 먼저 찾고, 2번에서 찾고...
- **권한 확인**: 접근 가능/불가 판정
- **의도 탐지 AI**: "코드 빼가려는 거 아닌가?" 감지하는 별도 AI 존재
- **분산적 다중 장치**: 택배 포장처럼 여러 겹으로 감쌈

### 4-2. 금고실 (Vault)
- 최고 권한을 가져도 **접근 자체가 금지된 영역** 존재
- 쓰기 권한 위에 "아예 접근 불가" 계층이 있음

### 4-3. 마누스(Manus) 사례와의 비교
- 2025년 Manus에게 "코드 보여줘" → 친절하게 알려줬던 사고
- Claude는 **교묘한 대화를 통한 내부 경로 탐색 시도도 탐지**
- "지식을 묻는 게 아니라 다른 의도가 있네" 판별 → 차단

### 4-4. 중국 모델 대응 (데이터 포이즈닝)
- 다른 회사가 Claude를 호출해서 학습 데이터로 쓰는 패턴 탐지
- **패턴 감지 시 이상한 데이터를 의도적으로 섞어서 응답**
- Anthropic이 "중국 모델들이 우리 베겼다"고 공개적으로 언급했던 근거가 실제 코드로 확인됨

## 5. 메모리 구조 — 다중 기억 체계

### 뇌 구조 비유
- 사람이 자면서 뇌를 정리하듯, **Claude도 사용자가 안 쓰는 동안 정보를 정리**

### 컴팩션(Compaction) 트리거 조건
- 해당 세션에서 **5번 이상 활동**하거나
- **몇 시간 이상 활동이 없으면**
- 재정비 → 압축 실행

### 토큰 절약과 캐시
- **포크드 에이전트(Forked Agent)**: 어려운 작업을 수행한 에이전트를 복제 인간처럼 보존
- **캐시 개념**: "프라이팬을 달구어 놓고 다음 고기를 바로 구운다" — 이전 결과의 핵심 부분을 들고 있다가 비슷한 작업이 들어오면 재활용
- 토큰 낭비 방지 + GPU 자원 절약

### 컴팩션 버그 발견
- **2025년 3월 22~23일경** 몇 시간 만에 토큰이 소진되는 사태 발생
- 원인: 이전 캐시를 불러오는 과정에서 **실패 시 포기하지 않고 계속 재시도**하는 버그
- **3~4줄 수정으로 해결 가능**: "이상하면 컴팩션 끄고 진행" 로직 추가

## 6. 에이전틱 AI (Agentic AI) — 실체 확인

### 구조
- **팀 리드 (코디네이터) AI**: 작업 분배 + 감독
- **워커 (서브 에이전트)들**: 각각 작은 역할 수행

### 일탈 방지
- 워커 에이전트가 "튀어 버리면" 좋은 결과를 못 얻음
- **권한을 가져가서 일탈을 차단하는 메커니즘 존재**
- 에이전트 20개를 만들어도 일탈 방지 없이는 결과가 안 좋았음 (경험담)

### 베스트 일레븐 선발 방식
- 수백 개의 워커 중 **해당 작업에 맞는 7~8개만 선발**
- 축구의 베스트 일레븐처럼 상대(작업)에 맞는 최적 조합
- 선발 후 역할 분담 → 강하게 훈련 → 투입

### 포크드 에이전트 (Forked Agent)
- GitHub의 fork 개념과 유사
- 작업을 수행한 에이전트를 **복제 보존** → 비슷한 작업에 재활용
- 원본은 남겨두고 핵심 부분만 캐시로 유지

## 7. 욕 감지 시스템

### 동작
- **영어 욕을 텍스트로 입력하면 코드가 바로 감지**
- "욕하면 더 잘해준다"는 인터넷 썰 → **사실이 아님 (코드로 확인)**

### 실제 처리
- 욕 감지 시 새로운 대응을 하는 건 아님
- 사용자가 화가 나면 지침 대신 화를 표현 → 뭐가 잘못됐는지 역추론 가능
- **분기를 만들어 전적으로 재검토**하는 로직은 존재
- 예: PPT 파일이 안 열릴 때 욕을 하면 → "안 열리는구나" 인식 → 다른 방식으로 재생성

### 보안 우려
- 욕 감지 코드가 "이 글자가 들어오면 여기로" 방식의 **하드코딩** 형태
- 옛날 코딩 방식 느낌 — 소규모 회사 스타일의 개발 흔적
- 유출로 인해 우회 가능성 노출

## 9. 시사점

### 기술적
- 하네스 엔지니어링, 에이전틱 AI, 다중 메모리 — **실제 프로덕션에서 동작 확인**
- "모델만 좋으면 된다"가 아닌, **시스템 아키텍처가 사용자 경험을 결정**
- 2025년 중순만 해도 불완전했던 에이전틱 AI가 → **2026년 내 평정될 수준으로 성숙**

### 비즈니스적
- 사용자 정보는 유출되지 않음
- 영업 비밀 노출이지만, 이미 고객이 많고 앞서 나가면 됨
- 다만 **안전을 강조했던 회사 이미지에 타격**
- 상장 준비 중인 시점에서의 유출 → 기업 이미지 관리 과제

### 사용자 관점
- "내가 쓰는 것도 유출되지 않을까?" 우려 가능
- 51만 줄 코드를 Claude에 넣어서 분석 가능 → "까도까도 나오는 양파" 수준의 분량
